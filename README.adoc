----

sysctl -p

[source,bash]
----

----

[source,bash]
----
lspci | grep -i nvme
----

= CPU Architecture Compatibility & Building

== Overview

SPDK/DPDK uses CPU-specific optimizations that can cause compatibility issues when container images are built on different hardware than where they run. The most common issue involves advanced instruction sets like `VPCLMULQDQ` which may not be available on all target systems.

== Symptoms of CPU Incompatibility

If you see errors like these in your `spdk-tgt` container logs:
----
ERROR: This system does not support "VPCLMULQDQ".
Please check that RTE_MACHINE is set correctly.
EAL: unsupported cpu type.
[ERROR] Failed to initialize DPDK
----

This indicates a CPU instruction set mismatch between build and runtime environments.

== Build Strategies

=== Option 1: Build on Target Architecture (Recommended)

Build the SPDK container images directly on the same CPU architecture where they will run:

[source,bash]
----
# SSH to your target node
ssh user@your-k8s-node

# Clone project and build SPDK image
git clone <your-repo>
cd flint/spdk-csi-driver
docker build -f docker/Dockerfile.spdk -t your-registry/spdk-tgt:latest .
docker push your-registry/spdk-tgt:latest
----

*Advantages:*
- Optimal performance (uses all available CPU features)
- No compatibility issues
- Automatic hardware detection

=== Option 2: Generic Build for Wide Compatibility

Modify the Dockerfile to use generic x86-64 targeting:

[source,dockerfile]
----
# In docker/Dockerfile.spdk, replace the configure line with:
RUN export RTE_TARGET=x86_64-native-linux-gcc && \
    export RTE_MACHINE=default && \
    EXTRA_CFLAGS="-march=x86-64 -mtune=generic" \
    ./configure --with-ublk --with-rdma --disable-tests --disable-unit-tests && \
    make -j$(nproc)
----

*Advantages:*
- Works across all x86-64 systems
- Single image for all deployments
- Easier CI/CD pipeline

*Disadvantages:*
- 15-20% performance reduction
- Slower CRC32C operations (~3-4x slower)
- Reduced cryptographic performance

=== Option 3: Multi-Architecture Images

Build separate optimized images for different CPU families:

[source,bash]
----
# Generic build
docker build -f docker/Dockerfile.spdk.generic -t registry/spdk-tgt:generic .

# Optimized build (on target hardware)
docker build -f docker/Dockerfile.spdk -t registry/spdk-tgt:optimized .

# Use in deployment based on node labels
nodeSelector:
  cpu.feature/vpclmulqdq: "true"  # Use optimized image
----

== Recommendations

* **For Production**: Use Option 1 (build on target) for optimal performance
* **For Development**: Use Option 2 (generic build) for simplicity
* **For Mixed Environments**: Use Option 3 (multi-arch) with node selectors

|===

|Hardware AES acceleration
|Most modern instances

Communication Flow
Discovery Process
Node Agent → Local SPDK → SpdkDisk CRD → Controller
     ↓
1. Discover NVMe devices via lspci/SPDK RPC
2. Create/update SpdkDisk resources in Kubernetes
3. Controller watches SpdkDisk changes
4. Controller uses discovered disks for volume operations
Volume Operations
CSI Request → Controller → SpdkDisk CRDs → Volume Creation
                ↓
1. Controller receives CSI volume request
2. Queries available SpdkDisk resources
3. Selects optimal disks based on criteria
4. Creates RAID configuration using selected disks
Implementation Details
Node Agent Capabilities

Hardware Discovery: Direct access to PCI devices and sysfs
SPDK Integration: Local RPC communication with SPDK daemon
Resource Management: Creates/updates Kubernetes CRDs
Health Monitoring: Real-time device health and performance metrics

Controller Capabilities

Volume Orchestration: Manages volume lifecycle across nodes
Replica Management: Handles RAID configuration and rebuilds
Policy Enforcement: Applies placement and redundancy policies
Global Monitoring: Cluster-wide health and performance tracking
