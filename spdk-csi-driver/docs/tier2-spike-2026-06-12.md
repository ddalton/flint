# Tier-2 phase 7a spike — results

**Date:** 2026-06-12/13 (single session; decision basis in
`tier2-evaluation-2026-06-12.md`, design in `incremental-replica-rebuild.md` §7)
**Cluster:** runh + one c5d.4xlarge build/consumer node (`runh-aws-1781311110`)
**Artifacts:** `raid-skip-rebuild.patch` (6th carried patch, on SPDK v26.05),
`scripts/tier2-spike.sh`, image `dilipdalton/spdk-tgt:tier2-spike`
**Fixture:** 1 Gi `flint-r2` volume, plain RWO (no rejoin-bounce annotation —
the restart-intolerant class), writer doing timestamped fsync'd appends;
replica leg degraded by killing its node pod; catch-up chased it to `standby`.

## Verdict: GO for phase 7b, with two named design items

All four deliverables pass. The patch design needed one in-flight revision
(v2, below) — exactly the kind of finding the spike existed to surface — and
the window measurement says the <2 s p99 target needs the orchestrator to
pre-stage or parallelize the two NVMe-oF attach handshakes (7b design item),
not a faster SPDK primitive.

## Deliverable 1 — quiesce window

Full §7 sequence, scripted (12 cross-node RPCs through `kubectl exec`,
~0.55 s harness round-trip each):

| step | ms |
|---|---|
| quiesce (leased) | 755 |
| snapshot E_f on survivor | 697 |
| export+attach E_f → dst | 2991 |
| esnap-clone head on dst | 801 |
| export+attach head → consumer | 2832 |
| lease renew (gate) | 842 |
| add --skip-rebuild | 647 |
| unquiesce | 715 |
| **total window / guest stall** | **10280 / 9510 measured** |

~6.6 s of that is harness exec overhead. The irreducible window with the
current sequence is ≈3.2 s: two NVMe-oF TCP attach handshakes (~1.2 s each)
plus six metadata ops (snapshot, clone, ns/listener adds, add) at
~50–300 ms. **To beat 2 s, 7b should pre-create the subsystem/listener
skeletons outside the window and/or run the two attaches concurrently** —
the data dependencies allow it (only snapshot→clone→add must be serial).
SPDK-side primitive costs are not the bottleneck.

## Deliverable 2 — correctness scrub

With the writer running throughout the rejoin and after: re-quiesced,
snapshotted both legs in the same window, exported both, md5 from the
consumer's host kernel over nvme-tcp:

```
79843803e94e0c19981ea81cc321352f  survivor leg (scrub-a)
79843803e94e0c19981ea81cc321352f  rejoined esnap-clone leg (scrub-b)
```

Bit-identical. The single-window invariant (snapshot and add under one
quiesce) is what guarantees this — see the v2 design note.

## Deliverable 3 — crash drills

- **Orchestrator death mid-window** (formal drill + several real script
  deaths during development): the lease auto-released every single time.
  Measured stalls: 7.96 s on an 8000 ms lease, 14.98 s on 15000 ms —
  guest io resumes with no operator action; `bdev_raid_quiesce_list`
  confirms the table empties. A renewal mid-window correctly extends
  (28.23 s = renew + 20 s observed).
- **Breached window**: with v1, a lease that expired mid-window let the
  add proceed against a stale clone (silent divergence — observed live as
  a leg missing a seconds-wide write gap; unwound by detaching the
  controller). v2 closes this twice over: the RPC refuses a skip_rebuild
  add with no held lease (-EPERM), and the spike script's pre-add renew
  aborts the sequence if the lease is gone.
- **Crash between add and record flip**: the spike never flips the
  record, so the post-rejoin state IS this crash state. A full
  unstage/restage converged to the record: the foreign leg was discarded,
  the real standby was admitted (Tier-1 admission), the filesystem
  mounted clean and all pre-restage data was intact. Rev-5's
  record-as-sole-authority contract holds; a half-finished hot rejoin
  costs a redundant catch-up, never corruption or stale reads.

## Deliverable 4 — patch pipeline

Sixth patch in `Dockerfile.spdk`; applied and compiled clean on v26.05
twice (v1, v2) on the remote build node (~12 min full image rebuild on
16 vCPU). The v1→v2 iteration is a live rehearsal of the maintenance
reality the evaluation priced in: this patch must be re-validated
*behaviorally*, not just "applies cleanly", on every SPDK bump.

## The v2 design revision (found live, hour one)

v1 copied the rebuild-finish idiom: the add took its own bdev quiesce
around channel installation. Live, that nested quiesce **pends behind the
orchestrator's held lease** (overlapping module quiesce ranges conflict
at the bdev layer), hanging the add; worse, identical (0, blockcnt)
ranges are aliased by `spdk_bdev_unquiesce`'s first-exact-match pop, so
the add's internal unquiesce could release *the lease's* range. v2:
channel installation runs inside the caller's held lease (no nested
quiesce), and the RPC enforces lease-held as a precondition for
skip_rebuild. The §7 "conservative ordering" footnote is thereby
resolved: the single window isn't a convention, it's checked.

Unexplained residual from v1 (not reproduced under v2): `lease_find`
returned ENOENT while the lease's expiry poller demonstrably remained
armed and fired on schedule. v2 ships `bdev_raid_quiesce_list` and
thread-stamped lease lifecycle logs; if it recurs, it will be directly
visible.

## Operational findings (7b backlog fodder)

- **Dead-leg controller reconnect flood**: after a replica node pod
  recreation, the consumer's old controller for the dead leg
  reconnect-loops against the rebuilt (re-fenced) export — `INVALID HOST`
  rejections ~100/s, flooding spdk-tgt logs past rotation within seconds
  (this hid the patch's NOTICELOGs entirely). A restage clears it. The
  node-agent reconcile should reap controllers whose subsystem now
  rejects them.
- **Tier-1/Tier-2 interplay is benign and useful**: any full restage
  converges the raid to the record — it admitted the caught-up standby
  (healing the volume) and discarded the spike leg. Hot rejoin and
  admission-at-reassembly compose without coordination.
- **Same-node reschedule reuses a dead staged mount** (known): after a
  consumer-node spdk-tgt roll, the writer needed a scale-to-zero cycle,
  not a pod bounce, to restage. The DS roll also reconfirmed that a
  consumer-side spdk-tgt restart without unstage kills the kernel
  initiator past `ctrl_loss_tmo` — the layered consumer fix recovers the
  SPDK side, but a mount that already went EIO needs the restage.

## What 7b orchestration signs up for (unchanged from the eval, plus)

Window sequence + unwind in the node agent/controller (unix-socket RPC
latencies make the 2 s target realistic), attach pre-staging or
concurrency, record flip after add (+ the new-leg lvol becoming the
replica's head: backfill epoch deltas, `set_parent` localization,
R_src-death → revert to stale), redundancy reporting until localization,
dead-controller reaping, and the §9-8 adversarial set run per SPDK bump.
